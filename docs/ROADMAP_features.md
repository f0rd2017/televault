# Roadmap: фичи, которых у нас нет (и которые реально добавить)

Сравнение с **Telegram-Drive (caamer20)** — Rust/Tauri-клиент. Их сильные стороны
проверены по исходникам. Здесь — что из этого реально встроить в нашу прогу
(`PySide6` + `Telethon` + `SQLite`, чанкованное+шифрованное хранилище, мультиаккаунт).

> ⚠️ Ключевое ограничение нашей архитектуры: файл у нас — это **набор
> зашифрованных чанков** (и/или batch-blob), а не одно telegram-сообщение.
> Поэтому всё, что у них «бесплатно» (нативная ссылка, стрим в любом клиенте),
> у нас требует **своего посредника**, который на лету собирает и расшифровывает
> части. Это цена нашей мощи (большие файлы, приватность, дедуп).

Светлая тема — **исключена по решению владельца**.

## Прогресс (реализовано)
- ✅ **Превью картинок** (скачанные + фоновая дозагрузка нескачанных).
- ✅ **Рекурсивный/глобальный поиск** (кнопка «Везде»).
- ✅ **Корзина (soft-delete)** — «В корзину», режим 🗑, восстановление, удалить навсегда / очистить.
- ✅ **Видео-постеры** — кадр скачанного видео через ffmpeg в `.thumb_cache` (фоном, не на UI-потоке; общая инфра превью из инкремента 1).
- ✅ **REST API** — локальный HTTP на stdlib (`app/api/`), выключен по умолчанию, токен-авторизация; чтение через repo, запись через `worker.submit_job`. См. `docs/REST_API.md`. Фундамент для будущих шеринга/стрима.
- ✅ **MTProto-прокси + резервный прокси с эскалацией** — `mtproto://…`/`tg://proxy?…` в дополнение к SOCKS5/HTTP; цепочка primary→backup→direct с переключением на лету (`proxy_backup`, `escalate_proxy`).
- ✅ **Throttle (лимит МБ/с)** — байтовый токен-бакет `BandwidthLimiter`, общий на инстанс; `upload_throttle_mbps`/`download_throttle_mbps` (0 = без лимита) в «Расширенных».
- ✅ **Шар-ссылки** — публичный `GET /share/<token>` на HTTP-сервере: сборка из чанков + раздача с HTTP Range (базовый стрим/перемотка). Пароль/срок опц. Таблица `shares` (schema v12), UI «Поделиться ссылкой». См. `docs/REST_API.md`.
- ✅ **Стрим без полного скачивания** — `GET /share/<token>` по Range качает и расшифровывает ТОЛЬКО перекрытые диапазоном части (а не весь файл). Plaintext-смещения частей выводятся из БД (`file_size − 32` для зашифрованных). `app/core/stream.py` (чистая раскладка/нарезка) + `downloader.fetch_parts_decrypted` (кэш частей, перемотка не качает заново) + `StreamResponse`/`_serve_stream`. Фолбэк на полную сборку для batch-member/неполных объектов.
- ✅ **Эвикция stream-кэша по объёму** — LRU-очистка `.share_cache/.stream` по таймеру (раз в 10 мин, `CacheManager.cleanup`); лимит настраивается: `stream_cache_max_mb` (дефолт 2048, 0 = без лимита) в «Расширенных». LRU-ключ — `max(atime, mtime)` (relatime не обновляет atime при чтениях; растущий префикс стрима не вытесняется первым).
- ⏭ Следующее (по аппетиту): транскод для не-нативных форматов; PDF-вьювер из облака на той же инфраструктуре.

## Tier 1 — дёшево, заметно (брать первыми)

### 1. Превью картинок в гриде
- **Что:** миниатюра вместо иконки для `*.jpg/png/webp/gif` (и кадр для видео — позже).
- **Зачем:** их `preview.rs` — одна из первых вещей, что бросается в глаза.
- **Как у нас:** в `app/ui/models_qt.py` (делегат `ExplorerGridModel`) для уже
  скачанных файлов рисовать `QPixmap` из `download_root`. Для нескачанных —
  фоновая дозагрузка только превью (хвост байт / уменьшенная версия) с кэшем в
  `cache_dir/.thumb_cache`. Лениво, как уже сделано для `local_presence`.
- **Оценка:** S (1–2 дня). Риск низкий, движок не трогаем.

### 2. Рекурсивный / глобальный поиск
- **Что:** поиск по имени по **всему** дереву, не только в текущей папке.
- **Зачем:** у них есть `cmd_search_global`; у нас поиск только внутри папки.
- **Как у нас:** `repo.list_objects_unified(folder=None, recursive=True, search=...)`
  уже почти всё умеет — добавить режим «искать везде» в строку поиска
  (`window_main.reload_items`) и быстрый индекс по `orig_name`.
- **Оценка:** S (1 день).

### 3. Корзина (мягкое удаление) перед безвозвратным
- **Что:** удалённое сначала в «Корзину» (логически), окончательная зачистка — отдельно.
- **Зачем:** защита от случайного удаления; у нас delete сразу бьёт по каналу.
- **Как у нас:** мы уже умеем логический tombstone (`deleted_ts`, `is_deleted`).
  Добавить вид «Корзина» (фильтр по tombstone) + «Восстановить» (снять флаг) +
  «Очистить корзину» (реальный `delete_remote`). Реюз существующих механизмов.
- **Оценка:** S–M (2–3 дня).

### 4. Ограничение скорости (throttle) загрузки/скачивания
- **Что:** лимит МБ/с на upload и download.
- **Зачем:** их `bandwidth.rs`/`vpn_optimizer.rs` — не дают забить канал.
- **Как у нас:** токен-бакет в `app/tg/upload/*` и `app/tg/download/*` вокруг
  отправки/чтения частей; настройка в диалоге (рядом с `concurrency`).
- **Оценка:** M (2–4 дня).

## Tier 2 — средне, высокая ценность

### 5. REST API поверх ядра
- **Что:** локальный HTTP API: список/загрузка/скачивание/статусы джоб.
- **Зачем:** их `api_routes.rs` (~1800 строк) — автоматизация и интеграции.
- **Как у нас:** `FastAPI`/`aiohttp` в отдельном потоке, тонкая обёртка над
  `DbRepo` (чтение) и `worker.submit_job` (запись). Ядро уже асинхронно-джобовое —
  ложится почти 1:1. Off by default, токен-авторизация.
- **Оценка:** M (3–5 дней).

### 6. MTProto-прокси (в дополнение к SOCKS5)
- **Что:** поддержка MTProto-прокси, не только SOCKS5.
- **Зачем:** у них есть; у нас только SOCKS5.
- **Как у нас:** Telethon поддерживает `connection=ConnectionTcpMTProxyRandomizedIntermediate`
  из коробки — прокинуть в `app/core/accounts.py` рядом с текущим прокси-резолвом.
- **Оценка:** S–M (1–2 дня).

### 7. Резервный прокси + escalation (primary → backup → direct)
- **Что:** второй прокси и переключение на лету при обрыве.
- **Зачем:** устойчивость к падению прокси посреди сессии (есть в нашем старом плане).
- **Как у нас:** см. отдельный план надёжности; `resolve_working_proxy` уже есть,
  нужно сделать цепочку tier'ов и `escalate_proxy(client)`.
- **Оценка:** M (3–4 дня).

### 8. Превью видео (кадр-постер) + базовая инфа (длительность/разрешение)
- **Что:** постер-кадр для видео в гриде + метаданные.
- **Зачем:** часть их преимущества по медиа без полного стрима.
- **Как у нас:** `ffmpeg`/`pyav` извлечь первый кадр после скачивания (или из
  скачанного префикса), положить в `.thumb_cache`. Без стриминга — дёшево.
- **Оценка:** M (2–4 дня, зависит от ffmpeg в окружении).

## Tier 3 — дорого / архитектурно (думать, а не хвататься)

### 9. Шар-ссылки (поделиться файлом по ссылке с паролем/сроком)
- **Что:** публичная ссылка на наш файл с паролем и сроком жизни.
- **Засада:** наши файлы зашифрованы и порезаны на части — «нативной» telegram-ссылки
  не существует. Нужен **свой мини-сервер-посредник**, который по токену собирает
  части, расшифровывает и отдаёт по HTTP (range). По сути — часть работы из п.10.
- **Как у нас:** тот же локальный HTTP-сервер (п.5) + таблица `shares` (токен,
  пароль bcrypt, expires, revoked) + сборка/расшифровка на лету.
- **Оценка:** L (1–2 недели). Полезно, но завязано на сервер-посредник.

### 10. Стриминг медиа без полного скачивания
- **Что:** играть видео/аудио прямо из облака.
- **Засада (двойная):** (а) нужен HTTP range-сервер; (б) наш AES-GCM шифрует
  **поблочно/по-частям**, так что произвольный seek = «дотянуть и расшифровать
  нужные части», а не байтовый рандом-доступ. Их случай проще: файл = одно
  сообщение, telegram сам отдаёт range.
- **Как у нас:** локальный сервер тянет только нужные части по запросу плеера,
  расшифровывает, отдаёт. Транскод (их `transcode.rs`) — отдельный большой кусок,
  можно начать без него (только «родные» форматы).
- **Оценка:** XL (несколько недель). Их главный отрыв; в лоб догонять невыгодно.

### 11. PDF-вьювер
- **Что:** просмотр PDF в приложении.
- **Как у нас:** `QtPdf` (есть в PySide6) поверх скачанного файла — средне; их
  «бесконечный скролл из облака» — дорого (тот же стрим-посредник).
- **Оценка:** M (вьювер скачанного) → L (потоковый из облака).

### 12. Кроссплатформенная упаковка + авто-апдейт + Android
- **Что:** готовые сборки и автообновление.
- **Засада:** Python/PySide6 пакуется тяжелее Rust/Tauri; Android для PySide6 —
  фактически нереалистично.
- **Как у нас:** `PyInstaller`/`briefcase` + GitHub Releases для desktop. Android — нет.
- **Оценка:** M (desktop-упаковка) / ✗ (Android).

### 13. i18n / мультиязычность
- **Что:** интерфейс не только на русском.
- **Как у нас:** вынести строки в `Qt Linguist (.ts)` или простой словарь-локаль.
  Объём механический, но большой (строки разбросаны).
- **Оценка:** M–L (нудно).

## Рекомендованный порядок
1. **Превью картинок** (п.1) — максимум «вау» за минимум работы.
2. **Рекурсивный поиск** (п.2) + **Корзина** (п.3) — закрывают очевидные пробелы UX.
3. **REST API** (п.5) — открывает интеграции и переиспользуется в шеринге/стриме.
4. **Throttle** (п.4) + **MTProto/резервный прокси** (п.6–7) — сетевые удобства.
5. Дальше — по аппетиту: видео-постеры (п.8), потом тяжёлые шеринг/стрим (п.9–10),
   опираясь на сервер из п.5.

> Стратегия: не гнаться за их медиа-стримингом в лоб. Сначала собрать **локальный
> HTTP-сервер один раз** (п.5) — и он же станет фундаментом для шеринга (п.9),
> стрима (п.10) и PDF-из-облака (п.11). Один кусок инфраструктуры — три фичи.
